## Scope: SPADE v1.6 — HTML rendering for scopes and plans (Pandoc)

**Intent:** Every locally-stored SPADE Scope and Plan gets a sibling HTML rendering that is genuinely pleasant to read in a browser — clean typography, status pills from frontmatter, table of contents, syntax-highlighted code, automatic dark mode, and a print stylesheet for PDF export. The markdown remains canonical (LLMs author it, PRs diff it, Linear receives it, `git grep` finds it); HTML is a **read-only rendered view**. When `/spade-scope` and `/spade-plan` write a file, they invoke the renderer and append a clickable `file://` link to the terminal so the user can cmd-click to open in their browser.

**Renderer choice:** Pandoc is a single recommended consumer binary (`brew install pandoc`, `apt install pandoc`, `winget install pandoc`). No vendored JavaScript bundle, no Node runtime, no compiled framework artefact, no npm install. Pandoc sits in the same dependency category as `git` or `jq` — a system tool the consumer has or installs. This keeps the framework's "Markdown + shell, no runtime" architectural posture intact; the ARCHITECTURE.md amendment is one line naming Pandoc as a recommended optional dep, not a license to introduce language runtimes.

**Minimum-viable v1.6.** This scope deliberately excludes Mermaid pre-rendering, read-skill terminal-link integration, bulk migration of historical `.md`, auto-injection into the consumer's `.gitignore`, and AC checkbox state. All are v1.7+ candidates once v1.6 usage data justifies them.

### Acceptance Criteria

1. **Renderer wrapper `~/.spade/bin/spade-render`.** A POSIX shell script (≤ 100 lines) that wraps a single `pandoc` invocation:
   - Interface: `spade-render <input.md> [--output <output.html>] [--stdout]`. Defaults to writing sibling `<input>.html`. Exit 0 on success, non-zero on failure. Prints absolute output path to stdout.
   - Pandoc invocation pins safety flags: `--from markdown-raw_html` (strips inline HTML — prevents `<script>` and event-handler injection), `--standalone`, `--embed-resources` (inlines the stylesheet and any referenced assets so the HTML is self-contained), `--metadata title:"$(extract from frontmatter)"`, `--template <SPADE template path>`, `--css <SPADE stylesheet path>`, `--toc --toc-depth=3`, `--highlight-style=tango` (syntax highlighting via Pandoc's built-in Skylighting; theme switches under `prefers-color-scheme: dark`).
   - **Graceful degradation:** if `pandoc` is not on `PATH`, the script prints to stderr `spade-render: pandoc not installed (https://pandoc.org/installing.html — recommended for HTML rendering of SPADE artefacts)` and exits 2. Callers (the skills) detect exit 2 and surface the install hint to the user on every invocation until pandoc is installed — **not** a one-time-per-session notice (adversarial-reviewer's "silent feature unavailability" finding).

2. **HTML template (`~/.spade/render/template.html`).** A Pandoc HTML5 template that produces:
   - A `<head>` with `<meta charset="utf-8">`, `<meta name="viewport">`, the inlined stylesheet via Pandoc's `--embed-resources`, and a **restrictive Content Security Policy** meta tag: `<meta http-equiv="Content-Security-Policy" content="default-src 'none'; style-src 'unsafe-inline'; img-src data:; base-uri 'none'; form-action 'none'">`. This prevents any `file://`-origin XSS escape from loading external resources, exfiltrating local files, or executing inline scripts — closes security-lens's `file://` trust-boundary finding.
   - A **status header** generated from frontmatter: coloured status pill (one of six SPADE phase colours documented in the stylesheet — `scoped`, `planning`, `approval`, `delivering`, `evaluating`, `done`), type pill, creation date, last-updated date, Linear link (if `linear_url` is in frontmatter), and the title as `<h1>`.
   - A **table of contents** auto-generated by Pandoc's `--toc` from `<h2>`/`<h3>` headings, rendered at the top of the document (not sticky — that's v1.7+ if useful; closes yagni's "sticky TOC is cargo-cult" finding).
   - Main content rendering all body markdown.
   - Frontmatter is consumed for the status header and not rendered as a code block in the body.

3. **Stylesheet (`~/.spade/render/spade.css`).** ≤ 12KB **uncompressed**, enforced by a CI shell check (`test "$(wc -c < spade.css)" -lt 12288`) that fails the framework's lint if exceeded. Provides:
   - System-font typography stack, comfortable line-length (60–80 chars), generous vertical rhythm.
   - Six status pill colours (one per phase) declared as CSS custom properties so a future v1.7 theming pass can override them in one place.
   - Skylighting-compatible code-block styling with a dark variant under `@media (prefers-color-scheme: dark)`.
   - Print stylesheet under `@media print` — hides the TOC, removes backgrounds, A4-friendly margins.
   - **No client-side JavaScript anywhere in v1.6.** Sticky TOC, interactive checkboxes, filterable tables, dark-mode toggle button — all deferred.

4. **Skill integration — writers only.** `/spade-scope` and `/spade-plan` SKILL.md prose updated so the write flow becomes:
   1. Write the `.md` file (existing behaviour, unchanged).
   2. Invoke `~/.spade/bin/spade-render <path>.md` via `Bash`.
   3. Print a closing line in the exact form `View in browser: file://<absolute-path>.html`. The `file://` URL is absolute (auto-linkified by iTerm2, Warp, VS Code, Terminal.app). On render failure (exit 2 = pandoc absent), print instead `(HTML render unavailable: install pandoc — https://pandoc.org/installing.html — to enable. .md at <path>)`. Skills never abort on render failure; `.md` is the canonical artefact.
   - **Mode-resolver interaction (M-879):** in `local` and `hybrid` modes the skill writes a local `.md` and invokes the renderer. In `linear` mode the skill does not write a local `.md` (per M-879 AC#5), so no render and no terminal link — closes architecture-strategist's "five of seven skills ambiguous" finding.
   - **Read-skill integration (`/spade-status`, `/spade-list`, `/spade-approve`, `/spade-evaluate`, `/spade-review`) is explicitly out of scope for v1.6.** Users who want to open a rendered scope from a read skill can copy the path and run `spade-render` manually, or wait for v1.7.

5. **Security posture — explicit ACs.** Beyond the CSP meta in AC#2 and the `--from markdown-raw_html` flag in AC#1, the Plan must commit to and CI-verify:
   - **No raw HTML in body output:** a fixture file `tests/fixtures/render/xss-attempts.md` containing `<script>alert(1)</script>`, `<img src=x onerror=alert(1)>`, `[link](javascript:alert(1))`, and `{onclick=alert(1) .pill}` (markdown-attr-style — Pandoc does not enable this by default, but the test pins it). The rendered HTML MUST contain none of those payloads in executable form. CI grep asserts: no `<script`, no ` on[a-z]+=`, no `javascript:` URL in any rendered fixture output.
   - **No filesystem paths in body output:** a CI grep asserts the rendered HTML contains no `/Users/`, no `/home/`, no host-specific strings anywhere. Frontmatter-derived absolute paths (if any) are excluded by the template.
   - **CSP meta present on every render:** CI grep asserts every rendered fixture contains the exact CSP string from AC#2.

6. **`docs/FRAMEWORK.md` §HTML Rendering.** New section ≤ 60 lines covering: Pandoc as a recommended dep with install commands per OS, the renderer's role (view, not data), `spade-render` invocation syntax, the status pill palette (six hex/HSL values inline so AC#2 has a single source of truth), the CSP/security stance, the cross-platform `file://` URL construction rule, the recommended `.gitignore` line for `.spade/**/*.html` (text only — **not** auto-injected, closes yagni's gitignore-injection finding), and graceful-degradation behaviour. Cross-referenced from a one-line addition to `ARCHITECTURE.md` §External-Toolchain-Policy naming Pandoc as a recommended consumer binary in the same category as `git` or `jq`. **PATTERNS.md is not modified** — Pandoc is a system tool, not a library dep, so "Markdown + YAML is the only data format" stands unamended.

7. **Release packaging.** v1.6.0:
   - `VERSION` set to `1.6.0`.
   - `CHANGELOG.md` entry covering: Pandoc-based rendering, terminal `View in browser:` link on `/spade-scope` and `/spade-plan`, security posture (CSP + `--from markdown-raw_html`), the FRAMEWORK section, the ARCHITECTURE one-line amendment.
   - `/spade-update` v1.5.x → v1.6.0 migration recipe: checks for `pandoc`, reports presence/absence, **does not bulk-render historical `.md`** (lazy on next write only — closes security-lens's "bulk render of attacker-influenced content" finding and yagni's "value lazy render already provides" finding).
   - `CLAUDE-section.md` consumer fragment unchanged (no skill-table row; the renderer is invoked by existing skills, not a new slash command). Fragment version stamp bumps to 1.6.0.

### Architectural Constraints

- **Pandoc is a recommended consumer dep, not a library.** It joins `git`, `jq`, and the shell as a system tool. ARCHITECTURE.md §External-Toolchain-Policy is amended by one line naming this. PATTERNS.md "Markdown + YAML frontmatter is the only data format" is preserved verbatim — HTML is a **rendered view**, not data the framework reads back.
- **No vendored third-party code** — no JS bundle, no npm, no compiled artefacts. The renderer is a POSIX shell wrapper.
- **No client-side JavaScript** in any rendered HTML in v1.6. Pure CSS for everything.
- **Stylesheet inlined per-file** via Pandoc's `--embed-resources`. ~12KB overhead per HTML; portable, self-contained, no broken styles when copied.
- **Determinism within a Pandoc minor version.** Identical `.md` produces identical `.html` when rendered by the same Pandoc minor version. Across minor versions, output may differ; this is documented in FRAMEWORK.md (closes adversarial-reviewer's "byte-identical for the lifetime of v1.x is not a promise the maintainer can keep" finding).
- **Render failure never aborts a skill.** `.md` is the canonical artefact; HTML is convenience.
- **Per-write rendering only.** No bulk migration on `/spade-update`; no lazy re-render on read skills (read-skill integration is out of scope for v1.6).
- **Cross-platform `file://` URLs.** Renderer prints platform-correct URLs (`file:///Users/...` on macOS/Linux, `file:///C:/...` on Windows). Documented in FRAMEWORK.md.

---

### Dependencies

- **Pandoc 3.0+** as a recommended consumer dep. Graceful degradation when absent (AC#1 / AC#4). `/spade-onboard` is **not** modified to install Pandoc automatically (v1.6 out of scope); operators install it themselves per the FRAMEWORK doc instructions.
- No npm, no Node, no Python (beyond what SPADE already uses for CI lint), no vendored bundles, no compiled artefacts on either the framework side or the consumer side.

### Context

- **Upstream:** v1.4 (M-879) defined `.spade/scopes/<slug>.md` and `.spade/plans/<scope-slug>-plan.md` as the canonical local layout. v1.6 generates sibling `.html` at those exact paths in `local` and `hybrid` modes; no rendering in `linear` mode (the artefact lives in Linear).
- **Downstream:** Consumer repos pick up via `/spade-update` v1.5.x → v1.6.0 fragment re-stamp. No bulk render on upgrade.
- **Related:** v1.5 (M-889, /spade-unhinged) is orthogonal — unhinged sessions don't write scopes/plans so no rendering happens. AC#4 explicitly confirms `/spade-unhinged` emits no terminal link (closes architecture-strategist's nit).

### Out of Scope

- **Mermaid diagram rendering** (any form — pre-rendered via `mmdc`, client-side via JS, or otherwise). v1.7+ candidate once a real scope contains a Mermaid block worth rendering. Cut per yagni-simplicity, security-lens, and adversarial-reviewer convergence; defers ~300MB Chromium toolchain.
- **Read-skill terminal links** (`/spade-status`, `/spade-list`, `/spade-approve`, `/spade-evaluate`, `/spade-review`). v1.7+ candidate. Users who want to open a rendered scope from a read skill can run `spade-render` manually or open the sibling `.html` directly.
- **Bulk render on `/spade-update`** for historical `.md`. Lazy render on next write only.
- **Auto-injection of `*.html` into consumer `.gitignore`.** v1.6 documents the recommended line in FRAMEWORK.md and lets consumers add it explicitly (yagni's finding).
- **Sticky TOC**, **interactive checkboxes**, **filterable findings tables**, **dark-mode toggle button**, **client-side JavaScript of any kind**. v1.7+ if real demand surfaces.
- **AC checkbox state persistence** (localStorage or otherwise). HTML is read-only.
- **`.spade/learnings/` HTML rendering.** Learnings are short and fine as `.md`. v1.7+ if useful.
- **Panel-report HTML rendering** (`/spade-review` output as a filterable HTML report). Genuinely valuable but needs JS for filtering. v1.7+.
- **GitHub Pages or static-site generation.** Local-first only.
- **Theming or per-repo stylesheet overrides.** Single ship-with-framework stylesheet. v1.7+ may add `.spade/.spade-render-config.json`.

### Origin

Ad-hoc — user request 2026-05-13 to make SPADE Scopes and Plans easier to read by saving them as HTML files with clever formatting. After a 5-persona panel review (33 findings on the v1 draft, 41 on the v2 draft including 4 blocking findings on the original Node-bundle architecture), the scope was rewritten around Pandoc as the renderer and stripped to minimum-viable shape (5 ACs covering renderer + template + stylesheet + skill integration + security + docs + release).

### Risk / Unknowns

- **Pandoc not installed.** Most engineering machines have it but not all. Graceful degradation works; the install hint surfaces on every write until installed (per AC#1). `/spade-onboard` could probe for it in v1.7 if usage shows operators forget.
- **Pandoc version skew.** Across minor versions, output may differ slightly. Documented in FRAMEWORK.md; consumers who commit HTML may see small diffs on Pandoc upgrades. Acceptable tradeoff for not vendoring.
- **CSP false positives.** A restrictive CSP could block legitimate content (e.g. an embedded image). v1.6 stylesheet uses no external images; if v1.7 adds Mermaid SVG, the CSP needs updating (documented).
- **`file://` linkification across terminals.** iTerm/Warp/VS Code/modern Terminal.app: yes. tmux/screen/CI runners: variable. The link is plain text either way — copy-paste works as a fallback. Documented in FRAMEWORK.md.
- **`pandoc --from markdown-raw_html` excludes inline HTML entirely** — including legitimate uses like `<details>`/`<summary>`. v1.6 scopes/plans don't use raw HTML today; if a future scope wants `<details>`, it ships as a v1.7 addition to the template, not as an unbounded raw-HTML allowance.

### Delivery Preference

Mostly AI-delivered — shell wrapper, Pandoc template (small), CSS stylesheet, two SKILL.md edits, one FRAMEWORK section, one ARCHITECTURE line, CHANGELOG, VERSION bump, `/spade-update` recipe. Manual verification: human runs the upgraded `/spade-scope` against this repo, opens the rendered HTML in three browsers (Chrome, Safari, Firefox), confirms status header + TOC + dark mode + print preview look right, runs the XSS fixture suite to confirm AC#5 holds, and runs the renderer without Pandoc installed to confirm graceful degradation.
